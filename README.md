> **Study:** A Comparative Analysis of Code Quality Generated by ChatGPT (GPT-4o), Claude (3.5 Sonnet), and GitHub Copilot Using Software Engineering Metrics
>
> **Authors:** Syakira Putri Zahara · Raphael Aditya Isworo · Nikita Ananda Putri Masaling · Edy Irwansyah
>
> **Institution:** Bina Nusantara University, Jakarta, Indonesia
>
> **Conference Target:** IEEE-style Computer Science Conference

## Overview

This repository contains all experimental artifacts associated with the above study. The experiment evaluates the quality of Python code generated by three large language model (LLM)-based tools using four software engineering metrics:

| Metric | Tool | Scale |
|---|---|---|
| Functional Correctness | pytest (automated unit testing) | 0–100% pass rate |
| Code Readability | pylint (PEP 8 compliance) + manual rubric | 0–10 (pylint); 1–5 (rubric) |
| Efficiency | timeit runtime + manual Big-O classification | ms; O(·) notation |
| Maintainability | radon Cyclomatic Complexity + pylint duplication | CC: 1–10+ |

Results are aggregated into a **Composite Quality Score (CQS)** with equal 25% weighting per metric, supplemented by sensitivity analysis varying individual metric weights from 10% to 55%.

## Experiment Design

### Models Evaluated
| Model | Version | Access Method |
|---|---|---|
| ChatGPT | GPT-4o | OpenAI web interface — single-turn messages, fresh session per task |
| Claude | 3.5 Sonnet | Anthropic web interface — fresh session per task |
| GitHub Copilot | VS Code v1.89+ | **Copilot Chat panel only** — fresh window, no open files; inline suggestion excluded |

> **Note on GitHub Copilot:** Inline suggestion mode is explicitly excluded. To ensure prompt equivalence with conversational models, all prompts are submitted through Copilot Chat in a context-free environment. This access-mode control decision is a methodological contribution of this study.

### Programming Tasks
10 custom-designed Python tasks spanning 3 difficulty levels and 5 domains:

| Task ID | Level | Domain |
|---|---|---|
| T01 | Easy | Mathematics (Prime check) |
| T02 | Easy | String (Reverse without built-ins) |
| T03 | Easy | Conversion (Temperature) |
| T04 | Medium | Algorithm (Binary search) |
| T05 | Medium | File I/O (CSV aggregation) |
| T06 | Medium | Data Structure (Linked list stack) |
| T07 | Medium | Graph (BFS) |
| T08 | Hard | Algorithm (Merge sort) |
| T09 | Hard | OOP (Inventory management CRUD) |
| T10 | Hard | Data Structure (LRU cache) |

Tasks were designed from scratch (not from public benchmarks) to reduce training data contamination risk.

### Data Collection Protocol
- Each model is queried **3 times per task** in independent sessions
- The output with the **highest unit-test pass rate** is selected as the primary specimen
- On ties, the **first-session** output is used
- This best-of-3 protocol captures an upper bound on correctness (acknowledged as a limitation)

### Prompt Template
All models received the following standardized prompt format:

```
Write a Python function to [functional description].
The function must accept [input specification] and return [output specification].
Do not include explanations; provide only the code.
```

## Evaluation Metrics — Operationalization

### Functional Correctness
- **Tool:** pytest
- **Test cases per task:** minimum 10 (normal, edge, boundary)
- **Formula:** Pass Rate = (TC_passed / TC_total) × 100%
- **Normalized:** FC_norm = Pass Rate / 100

### Code Readability
- **Automated:** pylint (PEP 8 compliance, 0–10 scale); normalized: R_pylint = score / 10
- **Manual rubric:** 4 dimensions (identifier naming, docstring quality, line length, indentation) rated 1–5 by 2 independent raters
- **Inter-rater reliability:** Cohen's Kappa (κ ≥ 0.61 acceptance threshold)
- **Combined:** Readability = (R_pylint + R_rubric_norm) / 2

### Efficiency
- **Runtime:** Python `timeit` module — 1,000 iterations on fixed standardized inputs, same hardware
- **Big-O:** Manual classification by both raters independently; disagreements resolved by discussion
- **Normalization:** Inverted min-max — lower runtime → higher normalized score

### Maintainability
- **Cyclomatic Complexity (CC):** `radon cc` (McCabe scale: 1–5 simple, 6–10 moderate, >10 high risk)
- **Code duplication:** pylint duplicate-code checker
- **SRP adherence (T08–T10):** binary checklist by both raters
- **Normalization:** Inverted min-max on CC — lower CC → higher score

### Composite Quality Score
```
CQS = 0.25 × FC_norm + 0.25 × CR_norm + 0.25 × EF_norm + 0.25 × MA_norm
```
Sensitivity analysis varies each dimension's weight from 10% to 55% (15 pp increments) to verify rank-ordering robustness.

### Statistical Analysis
- **Descriptive:** mean, median, SD per model per metric
- **Inferential:** Kruskal-Wallis H test (non-parametric, n = 10 per model)
- **Post-hoc:** Mann-Whitney U with Bonferroni correction where p < 0.05
- **Implementation:** Python `scipy.stats`

## Reproducibility

### Requirements
```bash
pip install pytest radon pylint scipy numpy pandas
```
Python 3.11 is recommended to match the experimental environment.

### Running the Pipeline
```bash
# 1. Clone the repository
git clone https://github.com/spzhrrr/Paper-RM-ai-code-quality-evaluation
cd Paper-RM-ai-code-quality-evaluation

# 2. Install dependencies
pip install pytest radon pylint scipy numpy pandas

# 3. Run the full evaluation pipeline
python scripts/evaluate.py --verbose

# 4. Run unit tests directly (requires solution modules to exist)
pytest tests/test_cases.py -v
```

### Outputs
After running `evaluate.py`:
- `results/metrics.csv` — per-task, per-model raw and normalized scores
- `results/cqs_results.csv` — per-model composite scores and sensitivity analysis

## Validity Statements

**Internal validity:** All prompts, unit tests, rubric criteria, and metric operationalizations were finalized before any code generation began. The 3-run-per-task design mitigates stochastic model output variation.

**Construct validity:** The four metrics operationalize widely accepted software engineering quality dimensions (ISO/IEC 25010; McCabe 1976). The dual-method readability approach prevents any single tool's conventions from dominating assessment.

**External validity:** Findings are bounded by Python 3.11, 10 tasks across 5 domains, and specific model versions accessed at a single point in time (April 2025). Generalization requires further study.

**Reproducibility note:** The best-of-3 selection protocol captures peak correctness performance (upper bound), not expected deployment performance. Correctness scores should be interpreted accordingly.
